# 单文件真实结果页设计说明

## 1. 背景

当前仓库的网页端已经具备：

- `audioqas/web/static/web-preview.html` 定义的单文件/对比页面结构
- `audioqas/web/static/web-preview-data.js` 中的展示映射与示例数据
- `audioqas/web/static/web-preview-app.js` 中的基础状态、渲染与上传调用
- `audioqas/web/api.py` 提供的单文件上传评测接口 `POST /api/evaluate/upload`

但当前单文件页面仍主要依赖静态示例内容。用户即使上传真实文件，页面也还没有把返回结果完整地映射进现有产品结构。

本设计说明只解决一个子项目：让单文件主流程在两个核心页面上显示真实结果。

## 2. 本次范围

本次仅覆盖以下两个页面的单文件主流程：

- `纯人声评测` 页的 `单文件测评`
- `综合音频分析` 页的 `单文件分析`

用户流固定为：

1. 用户进入对应页面
2. 选择一个本地文件
3. 前端调用现有上传接口
4. 页面进入处理中状态
5. 接口返回后，单文件结果区由真实结果填充
6. 如果失败，页面显示明确失败状态和错误提示

## 3. 明确不在本次范围内的内容

以下内容不属于这次实现：

- 历史记录持久化
- 导出能力
- 对比流程增强
- 新的批量产品流
- 页面整体重设计
- 后端接口重构
- 真正可感知的进度任务系统

后端现有批量接口可以保留，但本次不会在前端单文件流程中暴露或复用。

## 4. 推荐方案

采用“真实数据接管现有预览结构”的方案：

- 保留 `web-preview` 现有页面骨架与视觉层
- 保留现有 API 入口
- 在前端增加单文件运行态
- 用真实返回值替换当前单文件区域的静态示例内容

不采用“先做极简真实页再回填样式”的方案，因为会造成一轮额外 UI 重构。

不采用“先大规模抽象前端契约再接结果”的方案，因为当前第一步目标是把真实功能尽快落在现有预览上，而不是先做架构工程。

## 5. 前端架构设计

### 5.1 文件职责

- `audioqas/web/static/web-preview.html`
  - 保留结构壳和挂载点
  - 不继续承载业务规则

- `audioqas/web/static/web-preview-data.js`
  - 保存展示映射、列定义、标签文案、模型维度说明
  - 新增必要的真实结果展示辅助函数

- `audioqas/web/static/web-preview-app.js`
  - 保存页面状态
  - 调用上传接口
  - 将接口返回值映射为页面内部显示模型
  - 渲染单文件运行态

### 5.2 单文件运行态

前端按页面分别维护单文件运行态：

- 当前上传文件
- 请求状态：`idle / loading / success / error`
- 最近一次真实结果
- 最近一次错误信息

建议结构：

```js
runtimeState.single = {
  eval: {
    file: null,
    status: "idle",
    result: null,
    error: null,
  },
  analysis: {
    file: null,
    status: "idle",
    result: null,
    error: null,
  },
};
```

### 5.3 数据流

单文件流程的数据流为：

1. 触发文件选择
2. 调用 `/api/evaluate/upload`
3. 传入当前页面对应的 `domain` 与 `model_key`
4. 收到后端结果
5. 将后端原始结果映射为页面内部显示模型
6. 页面各区块只消费显示模型，不直接消费原始接口 JSON

这样可以避免 DNSMOS、NISQA、AudioBox、信号分析直接把各自结构散落到 DOM 逻辑里。

## 6. 显示模型设计

### 6.1 统一目标

前端需要为两个页面分别生成可稳定渲染的单文件显示模型，最少应包含：

- 文件摘要
- 结果概览
- 模型结果卡列表
- 信号分析卡列表
- 预处理追溯
- 详细数据表所需行数据

### 6.2 纯人声评测页

必须真实显示：

- 文件摘要
  - 文件名
  - 时长
  - 采样率
  - 声道
  - 当前模型
- 结果概览
  - 主分数
  - 总体等级/结论
  - 简短处理建议
- 模型结果卡
  - DNSMOS：`OVRL / SIG / BAK`
  - NISQA：完整展示其自身维度 `OVRL / NOI / DIS / COL / LOUD`
  - NISQA 不需要对齐为 DNSMOS 风格的 3 维卡片
- 信号分析卡
  - 至少显示 `LUFS / LRA / True Peak / Clipping / THD`
  - 如有 `SNR` 则显示
- 预处理追溯
- 详细数据表
  - `模型维度`
  - `信号分析`
  - `完整表格`

### 6.3 综合音频分析页

必须真实显示：

- 文件摘要
  - 文件名
  - 时长
  - 采样率
  - 声道
  - 当前模型
- 结果概览
  - 主分数
  - 总体等级/结论
  - 简短处理建议
- 模型结果卡
  - `PQ / CE / CU / PC`
- 信号分析卡
  - 至少显示 `LUFS / LRA / True Peak / Clipping / THD`
  - 如有 `Stereo` 则显示
- 预处理追溯
- 详细数据表
  - `模型维度`
  - `信号分析`
  - `完整表格`

### 6.4 缺失值规则

显示规则必须明确：

- 有真实值就显示真实值
- 后端没有返回的字段，不伪造新的业务含义
- 缺失值显示为 `--` 或等价空状态
- 当前阶段不把对比逻辑混入单文件流程
- NISQA 维度按模型真实输出完整展示，不为了和 DNSMOS 统一而删减指标

## 7. 错误态与加载态

### 7.1 加载态

上传开始后：

- 页面切到单文件场景
- 进度条进入处理中状态
- 保留当前页面与模型选择
- 不跳转到其他页面或场景
- 当前阶段使用阶段性伪进度，不承诺真实百分比
- 阶段文案可以按 `上传中 / 预处理中 / 模型评测中 / 信号分析中 / 整理结果` 递进

### 7.1.1 为什么不做真实百分比

当前后端接口是单次请求完成后一次性返回结果：

- 没有任务 ID
- 没有中间态查询接口
- 没有流式进度事件
- 当前评测 service 也没有进度回调机制

因此本阶段不把“真实进度”并入单文件结果真实化范围。

真实进度应作为下一阶段的独立子项目处理，届时再设计：

- 任务创建
- 进度上报
- 轮询或流式订阅
- 完成态取结果

### 7.2 成功态

上传成功后：

- 更新运行态为 `success`
- 用真实结果替换单文件区域中的静态示例
- 保留页面现有结构、说明卡和详细表格切换器

### 7.3 失败态

上传失败后：

- 更新运行态为 `error`
- 进度条显示失败
- 显示明确错误提示
- 不清空页面壳
- 不自动切换页面

## 8. 测试策略

本次实现至少需要补齐以下类型测试：

- Python 测试
  - 单文件页面结构引用与页面规则测试
- Node 测试
  - 数据映射函数测试
  - 详细数据列与真实显示模型兼容性测试

如果实现过程中引入新的纯函数映射层，应优先对该层写测试，而不是只做大段文本快照。

## 9. 成功标准

本次完成后，以下条件必须同时满足：

1. 用户在 `纯人声评测` 页上传真实文件后，页面能显示真实单文件结果
2. 用户在 `综合音频分析` 页上传真实文件后，页面能显示真实单文件结果
3. 结果页中的文件摘要、模型结果、信号分析、预处理追溯、详细数据表都来自真实返回值
4. 当当前模型为 `NISQA` 时，页面完整展示 `OVRL / NOI / DIS / COL / LOUD`
5. 页面保留当前 `web-preview` 的产品结构，不引入新的批量产品流
6. 上传失败时页面进入明确失败态
7. 测试通过
8. 当前阶段的进度展示为阶段性伪进度，而非真实百分比

## 10. 约束

- 必须遵守 `AGENTS.md` 中的边界：
  - `web-preview.html` 保持轻量
  - 业务展示逻辑集中到 `audioqas/web/static/web-preview-data.js`
  - DOM 状态与渲染逻辑集中到 `audioqas/web/static/web-preview-app.js`
- 不恢复旧桌面 App 叙事
- 不把本次工作扩展为历史页或对比页的大改造
