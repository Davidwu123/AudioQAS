# AudioQAS - 音频质量评测 macOS 应用需求文档

> 版本: v0.1 | 日期: 2026-05-14 | 作者: David

---

## 1. 产品定义

**名称**: AudioQAS

**定位**: 基于 DNSMOS 模型的音频质量评测工具，面向音频工程师、QA 团队和算法开发者，提供无参考 MOS 评分和可视化分析。

**目标平台**: macOS 13+ (Ventura 及以上)

**技术路线**: Python 3.9+ / PySide6 / ONNX Runtime / dnsmos 包

---

## 2. 用户角色与场景

| 角色 | 核心场景 |
|------|----------|
| 音频算法工程师 | 对算法处理后的音频批量打分，对比前后质量变化 |
| QA 测试人员 | 验收测试时批量评测音频样本，导出报告 |
| 研究人员 | 收集数据集的质量标注，统计分析 |

## 3. 功能需求

### 3.1 音频输入

| ID | 功能 | 优先级 | 说明 |
|----|------|--------|------|
| F-01 | 单文件分析 | P0 | 拖拽或文件选择器导入单个音频文件 |
| F-02 | 批量分析 | P0 | 支持导入整个目录或多个文件（上限 500 个） |
| F-03 | 格式支持 | P0 | WAV(PCM)、FLAC、MP3、AAC、OGG、M4A、MP4、MKV、AVI |
| F-04 | 自动预处理 | P0 | 非 16kHz 自动 resample，非 mono 自动转单声道 |
| F-05 | 时长校验 | P1 | 低于 3 秒的文件标注警告，仍可评分但提示结果可能不准 |
| F-06 | 录音评测 | P2 | 支持 Mac 内建麦克风实时录音并评分 |
| F-07 | 视频提取音频 | P0 | MP4/MKV/AVI 视频文件自动提取音轨，提取后与纯音频文件走相同评分流程 |
| F-08 | 视频格式元数据 | P1 | 视频文件评分结果中标注视频编码、帧率、音轨信息等元数据 |

### 3.2 评分引擎

| ID | 功能 | 优先级 | 说明 |
|----|------|--------|------|
| F-09 | DNSMOS 评分 | P0 | 输出 OVRL / SIG / BAK 三个维度分数（1.0~5.0），附通俗描述 |
| F-10 | 分段评分 | P1 | 长音频按固定窗口（如 10s）分段评分，展示时间维度变化 |
| F-11 | 参考对比 | P1 | 若同时提供原始和处理后音频，计算差值并标注改善/退化 |
| F-12 | 多模型评分 | P2 | 除 DNSMOS 外可选 NISQA（4维度+48kHz），结果并排对比 |
| F-13 | 模型选择栏 | P1 | 侧栏内嵌模型单选区，切换评分引擎，当前模型高亮标记，扩展新模型只需增加选项 |

### 3.3 结果展示

| ID | 功能 | 优先级 | 说明 |
|----|------|--------|------|
| F-13 | 分数卡片 | P0 | 单文件结果以卡片形式展示 OVRL/SIG/BAK，颜色编码+通俗描述+质量等级 |
| F-14 | 质量等级 | P0 | 分数映射到等级：Bad(1-2) / Poor(2-3) / Fair(3-4) / Good(4-4.5) / Excellent(4.5-5) |
| F-15 | 批量表格 | P0 | 列表视图，显示文件名、时长、三维度分数+通俗描述、等级，支持排序/筛选 |
| F-16 | 分段波形图 | P1 | 时间轴上叠加分数曲线，标注质量波动区域 |
| F-17 | 统计摘要 | P1 | 批量分析后展示均值/中位数/标准差/分布直方图 |

### 3.4 数据管理

| ID | 功能 | 优先级 | 说明 |
|----|------|--------|------|
| F-18 | 历史记录 | P1 | 本地保存评测历史，可回溯查看 |
| F-19 | 导出 CSV | P0 | 一键导出评分结果为 CSV，含通俗描述列 |
| F-20 | 导出 JSON | P1 | 导出完整结果含预处理信息（原始采样率、声道数等元数据） |
| F-21 | 清除历史 | P1 | 支持清除单条或全部历史记录 |

### 4.5 颜色编码规则

### 4.7 评分通俗描述映射

分数不仅要展示数值和等级，还需要用日常用语辅助解释，让非专业用户也能直观理解。

**OVRL (Overall) - 整体听感**

| 分数范围 | 通俗描述 | 场景比喻 |
|----------|----------|----------|
| 1.0-2.0 | 刺耳难听 | 像在嘈杂工地打电话，几乎听不清对方说什么 |
| 2.0-3.0 | 勉强能听 | 像在地铁里通话，有明显噪音但勉强能听懂 |
| 3.0-4.0 | 还行 | 像在办公室通话，有小杂音但不影响理解 |
| 4.0-4.5 | 清晰舒服 | 像在安静房间通话，声音清晰自然 |
| 4.5-5.0 | 非常清亮 | 像面对面交谈，声音纯净通透 |

**SIG (Signal Quality) - 语音清晰度**

| 分数范围 | 通俗描述 | 场景比喻 |
|----------|----------|----------|
| 1.0-2.0 | 模糊难辨 | 声音像隔着一层厚棉被，完全听不清 |
| 2.0-3.0 | 含糊不清 | 像水下的声音，需要费力才能听懂 |
| 3.0-4.0 | 基本清楚 | 语音大体清晰，偶尔有模糊的地方 |
| 4.0-4.5 | 清晰自然 | 像正常面对面说话，声音还原好 |
| 4.5-5.0 | 纯净透亮 | 录音棚级别的语音质量，细节完整 |

**BAK (Background Quality) - 背景干净度**

| 分数范围 | 通俗描述 | 场景比喻 |
|----------|----------|----------|
| 1.0-2.0 | 噪音轰鸣 | 背景像在酒吧里，噪音淹没了语音 |
| 2.0-3.0 | 噪音明显 | 背景像在繁忙马路旁，噪音较重 |
| 3.0-4.0 | 有些杂音 | 背景像在咖啡馆，有轻微杂音 |
| 4.0-4.5 | 安静清爽 | 背景干净，几乎无杂音 |
| 4.5-5.0 | 寂静纯净 | 隔音室级别，背景完全干净 |

UI 中分数卡片和表格均展示通俗描述（如 "3.82 / Fair / 还行"），CSV 导出也包含描述列。

### 4.6 交互细节

| 场景 | 行为 |
|------|------|
| 拖拽文件到窗口 | 直接进入评分流程，显示进度 |
| 拖拽目录到窗口 | 递归扫描音频文件，进入批量评分 |
| 评分进行中 | 状态栏显示进度条和计数，UI 不可关闭但可取消 |
| 评分完成 | 自动切换到结果视图 |
| 双击列表行 | 展开该文件的单文件详情视图 |
| 右键列表行 | 上下文菜单：查看详情 / 重新评测 / 导出单条 / 删除 |
| Cmd+S | 导出当前结果为 CSV |
| Cmd+E | 导出当前结果为 JSON |
| 文件时长 < 3s | 评分卡片旁显示警告图标和 tooltip 说明 |

### 5.2 存储位置

- 评测历史: `~/Library/Application Support/AudioQAS/history.db` (SQLite)
- 模型缓存: `~/Library/Application Support/AudioQAS/models/`
- 临时预处理文件: `~/Library/Caches/AudioQAS/preprocessed/`
- 配置文件: `~/Library/Preferences/AudioQAS/settings.json`

## 6. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 单文件评分 < 2s（10s 音频），批量 100 文件 < 60s |
| 并发 | 支持多文件并行评分，默认 4 并发，可配置 |
| 内存 | 峰值 < 500MB（批量 100 文件场景） |
| 安装包 | .app 体积 < 200MB（含 Python 运行时和 ONNX 模型） |
| 启动 | 冷启动 < 3s（模型预加载） |
| 兼容 | macOS 13+ (Intel + Apple Silicon) |
| 数据安全 | 不上传任何音频或评分数据，全部本地处理 |
| 无网络 | 评分功能完全离线可用（模型本地缓存后） |

| 4 | 批量上限 500 是否足够？ | 数据管理设计 | 500 以内用 SQLite，超限需分页/归档机制 |
| 5 | 深色模式适配范围？ | UI 工作量 | PySide6 跟随系统主题，自定义调色板覆盖 |

## 10. 评分模型对比与选择

### 10.1 DNSMOS 不是最准的模型

DNSMOS 是目前**使用最广泛**的无参考音频质量评分模型，但从学术评测数据看，它**不是最准确的**。以下是当前主流模型的对比：

| 模型 | 类型 | 采样率 | 输出维度 | 泛化能力 | 开源可用 | 适用场景 |
|------|------|--------|----------|----------|----------|----------|
| **DNSMOS** | 无参考 | 16kHz | OVRL/SIG/BAK | 中（域外下降明显） | 开源 | 噪声抑制评估 |
| **DNSMOS Pro** | 无参考 | 16kHz | OVRL/SIG/BAK | 较好（PCC 0.93 vs DNSMOS 0.89） | 微软内部 | DNSMOS 增强版 |
| **UTMOS** | 无参考 | 16kHz | MOS | 好（VoiceMOS 2022 冠军） | 开源 | 语音合成质量 |
| **NISQA** | 无参考 | 48kHz | MOS/NOI/COL/DIS/LOUD (5维度) | 较好 | 开源 | 实时通话全频带评估 |
| **SCOREQ** | 无参考/NMR | 16kHz | MOS | 最好（对比学习泛化最强） | 开源 | 跨域通用评估 |
| **NORESQA-MOS** | 非匹配参考 | 16kHz | MOS | 好（基准锚定） | 开源 | 有参考样本场景 |
| **SQ-AST** | 无参考 | 48kHz | MOS/NOI/COL/DIS/LOUD | 好（ITU-T 标准化候选） | 待发布 | 实时通话评估 |
| **SSL-MOS** | 无参考 | 16kHz | MOS | 视SSL模型而定 | 开源 | 研究用途 |

### 10.2 关键发现

1. **泛化问题是所有模型的通病** - DNSMOS 在训练域（噪声抑制）表现好，域外（语音合成、编解码）PCC 从 0.89 降到 0.53
2. **UTMOS 在 VoiceMOS 挑战中排名第一**，但同样受域外泛化影响
3. **SCOREQ 使用对比学习损失**，在 11 个跨域数据集上表现最稳定，是最有潜力的新一代模型
4. **NISQA 的独特优势**：支持 48kHz 全频带 + 5 维度评分（噪音、染色、断续、响度、整体），对 RTC 场景更有实用价值
5. **DNSMOS Pro** 是 DNSMOS 架构改进版（PCC 从 0.89 提升到 0.93），但微软尚未完全开源

### 10.3 对 AudioQAS 的建议

**M1 首选：DNSMOS** - 开源成熟、部署简单、社区支持最广，适合快速出产品

**后续迭代路线**：
- M3+: 加入 **NISQA** 作为可选引擎（48kHz + 5 维度对 RTC 行业更有价值）
- M5+: 评估 **SCOREQ** 替代 DNSMOS 作为默认引擎（泛化更好）
- 模型切换在设置页面的"评分模式"中实现，保持引擎抽象层

**技术实现要求**：评分引擎层必须设计为可插拔架构，不绑定单一模型。每个评分结果需记录使用的模型名称和版本。
