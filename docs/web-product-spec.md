# AudioQAS 网页端产品需求与操作指南

> 版本: v0.4
> 日期: 2026-05-19
> 适用范围: Web 端产品设计评审
> 对齐来源: `design/web-preview.html` + `design/web-preview-data.js` + `design/web-preview-app.js`

---

## 1. 产品目标

网页端不是在旧产品描述上做表面迁移，而是把当前音频质量分析能力整理成一个更清晰、更易理解、更适合持续扩展的在线工作台。

网页端当前目标有 6 个：

1. 让用户一眼分清两类任务场景：
   - `纯人声评测`
   - `综合音频分析`
2. 让用户在上传前就知道该走哪个页面，不要评测完才发现模型选错。
3. 让所有结果都既有 AI / MOS 判断，也有音频信号分析结果。
4. 为后续多文件、多组对比、任务历史、报告导出留出产品结构空间。
5. 让用户理解不同模型的适用场景、优缺点和分析原理，而不是只看到模型名字。
6. 让每次结果都可追溯：从原始文件到送入模型之前，经过了哪些预处理必须可见。

## 1.1 当前实现状态

当前仓库不是只保留静态预览，已经具备最小可运行网页端主链路：

- 页面入口：
  - `纯人声评测`
  - `综合音频分析`
  - `历史`
  - `设置`
- 真实上传链路：
  - `纯人声评测 / 单文件测评`
  - `纯人声评测 / 对比评测`
  - `综合音频分析 / 单文件分析`
  - `综合音频分析 / 对比分析`
- 真实结果映射：
  - 当前模型结果
  - 信号分析结果
  - 处理建议
  - 预处理追溯
  - 详细数据三视图
- 最小运行态页面：
  - `历史` 已从 `/api/history` 拉取列表
  - `历史` 已接入真实写入/读取链路，并在切页时主动刷新
  - `设置` 已接入默认模型显示、预处理追溯开关、默认对比模式联动
  - `设置` 已接入最小真实持久化链路，刷新页面后状态可恢复

当前文档仍允许描述后续扩展方向，但凡涉及当前产品边界判断，应以现有 `web-preview` 结构和当前真实实现为准，不以旧桌面形态或假交互为准。

## 2. 产品定位

AudioQAS 网页端是一个面向音频工程师、内容制作团队、QA 团队和审核人员的音频质量分析工作台。

它不是面向公众用户的轻量上传工具，而是偏专业、偏生产、偏审核的内部工具或团队工具。

因此网页端应具备以下气质：

- 工具感强于营销感
- 信息密度适中，但不追求极简
- 重点强调任务效率、对比效率、判断效率
- 对“上传前场景选择”和“结果解释”要足够明确
- 强调可追溯和可审视，不做黑盒式结果页

## 2.1 平台范围

网页端当前明确限定为：

- 仅支持 PC 浏览器
- 不支持移动端
- 不以手机和平板为目标交互设备

这意味着：

- 可以使用 hover 提示
- 页面布局优先按桌面宽屏设计
- 不需要为移动端单独设计复杂交互
- 响应式只需要覆盖不同 PC 宽度，不需要承担触屏主流程

## 3. 页面结构总览

网页端一级结构固定为 4 个页面：

1. `纯人声评测`
2. `综合音频分析`
3. `历史`
4. `设置`

其中前两个是核心工作页面。

### 3.1 纯人声评测

用于以下素材：

- 通话录音
- 口播录音
- 会议录音
- 纯人声采集
- 人声绝对主导、几乎无背景音乐的内容

默认能力：

- DNSMOS
- NISQA
- 音频信号分析

### 3.2 综合音频分析

用于以下素材：

- 人声 + 音乐
- 视频音轨
- 节目成品
- 混合内容音频
- 需要同时参考内容完成度和信号指标的内容

默认能力：

- AudioBox Aesthetics
- 音频信号分析

### 3.3 历史

用于查看：

- 历史任务列表
- 任务结果摘要
- 重新查看详情
- 重新导出
- 后续支持重新分析

### 3.4 设置

用于管理：

- 默认模型
- 预处理行为
- 导出偏好
- 运行策略
- 后续任务队列策略

## 3.5 模型说明与教育信息

网页端需要提供“模型说明区”，帮助用户理解：

- 这个模型适合什么场景
- 这个模型不适合什么场景
- 模型输出代表什么
- 模型分析原理大致是什么

这个说明不需要写成学术论文，但必须达到“业务用户能判断该不该用”的程度。

### 3.5.1 DNSMOS

需要说明：

- 适用场景：纯人声、通话、会议、口播
- 优点：
  - 对人声质量判断直观
  - 输出维度清晰
  - 适合快速筛查人声问题
- 局限：
  - 不适合复杂音乐或混合内容
  - 对强配乐、节目成品的判断参考意义有限
- 分析原理说明：
  - 基于无参考语音质量预测
  - 输出整体听感、语音清晰度、背景干净度

### 3.5.2 NISQA

需要说明：

- 适用场景：纯人声、高质量语音样本、需要更细分维度时
- 优点：
  - 维度更多
  - 对噪声、连续性、染色、响度有更细颗粒表达
- 局限：
  - 推理成本更高
  - 仍不适合强混合内容和节目成品
- 分析原理说明：
  - 基于无参考语音质量预测
  - 输出整体质量及多维语音感知维度

### 3.5.3 AudioBox Aesthetics

需要说明：

- 适用场景：人声+音乐、视频音轨、节目成品、混合内容
- 优点：
  - 更适合内容级判断
  - 能表达制作质量、内容享受、有用性、复杂度
- 局限：
  - 不是传统语音 MOS 模型
  - 分数不能和 DNSMOS / NISQA 直接横向等价比较
- 分析原理说明：
  - 基于内容美学 / 内容感知评分
  - 更接近“内容完成度与整体感受”，不是单纯人声通信质量

### 3.5.4 信号分析模块

也需要说明：

- 它不是 AI 模型替代品
- 它是与模型并行的工程判断层
- 用于发现响度、峰值、削波、失真等技术问题

### 3.5.5 模型说明的展示位置

建议采用两层：

1. 侧栏模型 hover 简介
2. 页面内“模型说明”折叠卡片

页面内模型说明卡片建议包含：

- 模型名称
- 一句话适用场景
- 优点
- 局限
- 分析原理简述

展示建议：

- 采用紧凑摘要卡，不建议写成长段说明
- 单张卡建议控制在 3 到 4 行信息结构内
- 推荐结构：
  - `适用`
  - `优点`
  - `局限`
  - `原理`
  - `指标`

指标解释建议：

- 模型说明区里增加一行 `指标` 解释缩写含义
- 例如：
  - `OVRL=整体听感，SIG=语音清晰度，BAK=背景干净度`
  - `PQ=制作质量，CE=内容享受，CU=内容有用，PC=制作复杂度`
  - `LUFS=综合响度，LRA=响度范围，True Peak=真实峰值，Clipping=削波次数，THD=谐波失真`

结果页展示建议：

- 卡片和表格里优先展示中文名称
- 缩写作为副标签保留
- 推荐形式：
  - `整体听感 · OVRL`
  - `综合响度 · LUFS`
  - `制作质量 · PQ`

详细数据表格规则：

- 表头的双语结构需要统一
- 不要出现有的列是两行、有的列是一行的混搭
- 推荐所有列都采用两层：
  - 第一层：中文主标签
  - 第二层：英文缩写或英文副标题
- 即使某列没有专业缩写，也建议补一个统一的英文副标题，例如：
  - `文件 / File`
  - `排序 / Rank`
  - `预处理追溯 / Pipeline`

不建议只显示缩写，否则首次使用者难以快速理解。

结果页规则补充：

- 结果页只展示“当前选中模型”的说明
- 不默认同时展开其他模型说明
- 信号分析模块可作为独立固定说明模块单独存在

## 4. 可观测性与日志模块需求

网页端当前已经具备真实上传、评测、对比、历史、设置等主链路。后续只靠页面现象判断“是否执行了”和“执行到了哪一步”已经不够，必须补统一日志模块，作为运行态排障和链路审计的基础设施。

日志模块目标不是机械记录所有函数，而是回答 4 个问题：

1. 当前请求是否真的进入后端主链路
2. 当前请求走到了哪个业务分支
3. 当前分支是否成功执行完成
4. 当前链路最终在哪一层中断或失败

### 4.1 日志目录与文件

- 日志目录固定为工程根目录下的 `log/`
- 主日志文件：
  - `log/audioqas.log`
- 错误日志文件：
  - `log/audioqas.error.log`
- 单个日志文件最大大小固定为 `20MB`
- 超过后必须滚动到新文件
- 滚动文件名采用后缀递增：
  - `audioqas.log`
  - `audioqas.log.1`
  - `audioqas.log.2`
  - `audioqas.log.3`
- `audioqas.error.log` 同样采用后缀递增规则

### 4.2 日志级别

日志级别必须在启动时可配置，不能靠改业务代码切换。

建议支持：

- `DEBUG`
- `INFO`
- `WARNING`
- `ERROR`

要求：

- 开发态默认 `DEBUG`
- 生产或上线环境可切到 `INFO` 或更高
- `ERROR` 日志同时写入：
  - `audioqas.log`
  - `audioqas.error.log`

### 4.3 统一日志格式

统一格式采用工程日志风格，并补足链路字段。

推荐格式：

```text
[2026-05-20 14:22:15.569][17505][I][req_compare_001][compare][model_score]:(dnsmos.py:136): [DNSMOS] score_succeeded group=A file=/tmp/a.wav ovrl=3.92 sig=4.10 bak=3.65
```

字段定义：

- `[2026-05-20 14:22:15.569]`
  - 时间戳，精确到毫秒
- `[17505]`
  - 线程号
- `[I]`
  - 日志级别缩写，使用 `D/I/W/E`
- `[req_compare_001]`
  - `request_id`
- `[compare]`
  - 当前场景，使用 `single / compare / batch / settings`
- `[model_score]`
  - 当前事件或步骤
- `:(dnsmos.py:136):`
  - 文件名与行号
- `[DNSMOS]`
  - 模块标签
- 后续正文
  - 使用 `key=value` 或简短自然语言混合表达

### 4.4 request_id 与链路范围

`request_id` 用于串起一次完整业务请求，不用于区分单个文件操作。

规则：

- 一次 `single` 请求对应一个 `request_id`
- 一次 `compare` 请求对应一个 `request_id`
- 一次 `batch` 请求对应一个 `request_id`
- 一次 `settings` 更新对应一个 `request_id`
- `compare` 内部无论上传多少文件或多少 group，仍共用同一个 `request_id`
- `batch` 内部无论上传多少文件，仍共用同一个 `request_id`
- 深层模块不得重新生成 `request_id`，必须透传

辅助字段：

- `scene`
  - `single / compare / batch / settings`
- `group_key`
  - 仅 compare 使用，如 `A / B / C`
- `file_path`
  - 当前正在处理的文件
- `model_key`
  - 当前模型

### 4.5 日志设计原则

日志设计不以“每个函数都打印 enter / exit”为目标，而以“可感知分支、可判断是否正确执行”为目标。

必须覆盖 4 类事件：

1. 链路事件
   - 例如：
     - `request_received`
     - `task_started`
     - `task_finished`
     - `request_finished`
     - `request_failed`
2. 分支事件
   - 用于记录走到了哪个分支以及为什么走这个分支
   - 例如：
     - `branch=video_extract`
     - `branch=resample`
     - `branch=mono_convert`
     - `branch=passthrough`
3. 结果事件
   - 用于记录关键输出摘要，而不是只写“done”
   - 例如：
     - 预处理输出文件路径
     - 模型主分数摘要
     - compare 排序结果
     - settings / history 写入结果
4. 异常事件
   - 用于记录失败位置、中断原因、异常摘要

### 4.6 关键打点层级

第一阶段至少覆盖以下层级：

1. API / 入口层
   - 接收请求
   - 参数摘要
   - 请求完成
   - 请求失败
2. Service / Task 编排层
   - 任务开始
   - 调用模型
   - 对比或批处理聚合
   - 任务结束
3. 模型执行层
   - 模型开始
   - 模型完成
   - 主分数摘要
   - 异常
4. 预处理层
   - 输入类型判断
   - 抽音 / 转单声道 / 重采样 / 直通分支
   - 生成的中间文件
5. Store / 持久化层
   - 设置读取
   - 设置写入
   - 历史写入
   - 文件损坏回退默认值

### 4.7 UI 无响应场景下的定位目标

日志模块必须支持“UI 点击后无反应”的全链路排障。

理想定位能力：

- 如果日志中没有对应 `request_id` 的 `request_received`
  - 说明问题大概率在前端事件、请求构造或请求发送之前
- 如果有 `request_received`，但没有后续 `task_started`
  - 说明问题在接口解析、参数校验或路由层
- 如果进入任务但卡在预处理
  - 说明问题在文件读取、视频抽音、重采样或中间文件生成
- 如果预处理完成但模型未完成
  - 说明问题在模型加载、推理或输入数据格式
- 如果后端已 `request_finished` 但 UI 仍无反应
  - 说明问题更可能在前端结果渲染或状态更新

因此日志模块不仅是记录系统行为，还承担“运行态全链路定位工具”的角色。

### 4.8 启动配置要求

日志模块必须支持启动配置项，至少包括：

- `AUDIOQAS_LOG_LEVEL`
- `AUDIOQAS_LOG_DIR`
- `AUDIOQAS_LOG_MAX_MB`
- `AUDIOQAS_LOG_BACKUP_COUNT`

要求：

- 本机运行入口统一初始化日志
- 所有模块统一从同一日志配置中心获取 logger
- 不允许每个模块单独配置 handler 或单独决定输出文件

### 4.9 第一阶段建议接入范围

第一阶段建议覆盖以下文件：

- `audioqas/web/run_local.py`
- `audioqas/web/api.py`
- `audioqas/web/tasks.py`
- `audioqas/web/services.py`
- `audioqas/core/preprocessor.py`
- `audioqas/models/dnsmos.py`
- `audioqas/models/nisqa.py`
- `audioqas/models/audiobox_aesthetics.py`
- `audioqas/models/analysis.py`
- `audioqas/web/history_store.py`
- `audioqas/web/settings_store.py`

## 5. 导航与场景区分规则

### 5.1 导航名称

导航中只显示主名称，不显示长副标题：

- `纯人声评测`
- `综合音频分析`
- `历史`
- `设置`

### 5.2 场景提示

导航 hover 时，展示提示：

- `纯人声评测`
  提示：`适用于通话、口播、会议、纯人声录音`
- `综合音频分析`
  提示：`适用于人声+音乐、视频音轨、节目成品与混合内容`

### 5.3 页面顶部说明

进入页面后，在页面顶部继续显示一句说明，避免信息只存在于 hover 中。

建议文案：

- `纯人声评测`
  页面说明：`面向通话、口播、会议、纯人声录音，结合人声质量模型与信号分析给出结果。`
- `综合音频分析`
  页面说明：`面向人声+音乐、视频音轨、节目成品与混合内容，结合 AI 内容评价与信号分析给出综合判断。`

补充规则：

- 页头标题和页头说明已经存在时，内容区不再重复放一组同义标题和同义副标题
- `历史` 和 `设置` 页面同样遵守这个规则

### 5.4 为什么不只做 hover 提示

原因如下：

- 移动端没有 hover
- 桌面端 hover 只适合“进入前提醒”
- 用户进入页面后仍然需要确认自己所在场景是否正确
- 页面顶部说明能降低误用成本

### 5.5 页面复用原则

`纯人声评测` 与 `综合音频分析` 除了模型输出结构不同，其他页面框架应尽量复用，不应做成两套完全独立产品。

应复用的部分包括：

- 顶部说明区
- 工具栏
- 上传区结构
- 文件信息区
- 预处理追溯区
- 信号分析区
- 导出区
- 对比输入区
- 历史结构

不完全相同的部分主要只有：

- 模型说明内容
- 模型输出卡片
- 综合结论文案

这样做的好处：

- 减少用户学习成本
- 减少设计系统分叉
- 后续实现更容易复用
- 结果页结构更统一

侧栏模型区补充规则：

- `纯人声评测` 页面只显示人声模型
- `综合音频分析` 页面只显示综合音频模型
- `历史` 与 `设置` 页面不显示模型选择区

## 6. 纯人声评测页面需求

## 6.1 页面目标

让用户快速完成纯人声素材的质量判断，并能在单文件和多组对比中快速定位问题。

## 6.2 页面组成

页面应由以下区域组成：

1. 顶部说明区
2. 工具栏
3. 上传区 / 空状态
4. 结果区
5. 对比结果区

内容区补充规则：

- 页头已经显示 `纯人声评测` 和场景说明后，内容区不再重复放一组同义标题
- 内容区直接进入上传区、结果区或对比区

## 6.3 工具栏

工具栏建议包含：

- `单文件测评`
- `对比评测`
- `导出结果`
- `重置`
- 进度显示

工具栏交互补充规则：

- 当前处于对比模式时，`对比评测` 按钮高亮
- 当前处于单文件模式时，`单文件测评` 作为主入口
- `单文件测评` 是功能入口，不应长期以“选中态/高亮态”表现

### 6.3.1 动态进度要求

进度不应只显示“处理完成”或静态状态，而应以动态形式逐渐变化，建议表现为：

- 百分比进度：`0% → 100%`
- 当前阶段提示
- 当前文件名

推荐展示形式：

- `预处理中 18%`
- `送入 DNSMOS 46%`
- `计算信号分析 78%`
- `结果整理完成 100%`

说明：

- 即使后端无法拿到真实细粒度模型进度，也应给出合理的阶段性渐进反馈
- 不能长时间停在单一静态状态
- 用户必须感知任务正在推进

当前阶段网页端不必在此处放“选择目录”作为强入口，目录导入和批量视图都不是一期重点，但结构上要预留后续扩展空间。

## 6.4 空状态

空状态应明确告诉用户：

- 这是纯人声页面
- 推荐上传什么素材
- 支持什么格式
- 可以直接开始单文件评测或进入对比模式

空状态文案建议：

- 标题：`拖拽纯人声音频到此处`
- 副标题：`适用于通话、口播、会议、纯人声录音`
- 支持格式：
  - `WAV / FLAC / MP3 / AAC / OGG / M4A`
  - `MP4 / MKV / AVI / MOV（如为纯人声视频可自动提取音轨）`

## 6.5 单文件结果

单文件结果至少包含：

1. 文件基础信息
   - 文件名
   - 时长
   - 原始采样率
   - 原始声道数
   - 是否预处理
   - 当前使用模型

2. 预处理追溯信息
   - 从原始文件到送模型前，经过了哪些预处理
   - 每一步预处理的目标和结果
   - 是否发生了视频抽音轨
   - 是否发生了重采样
   - 是否发生了转单声道

建议展示形式：

- `原始文件 → 抽取音轨 → 转单声道 → 重采样到 16kHz → 送入 DNSMOS`
- 或时间线 / 标签链形式

3. AI / MOS 结果
   - OVRL
   - SIG
   - BAK
   - 对应等级
   - 对应简短描述

4. 信号分析结果
   - LUFS
   - LRA
   - True Peak
   - Clipping
   - THD

5. 工程结论区
   - 是否适合直接交付
   - 是否需要降噪
   - 是否需要重做电平整理

### 6.5.1 单文件结果的信息优先级

推荐顺序：

1. 先看总体结论
2. 再看 MOS 卡片
3. 再看预处理追溯
4. 再看信号分析指标
5. 最后看详细解释

网页端不建议把所有指标平权平铺，首屏需要有明显主次。

单文件首屏布局建议：

- 不建议把 `Result Overview` 放在只占半宽的单卡里
- 更建议使用一张全宽总览卡，再在卡内分成两栏
- 左侧放：
  - 文件名
  - 总分 / 等级
  - 首屏结论
- 右侧放：
  - 文件摘要
  - 工程判断
  - 预处理追溯

这样可以保证“总体判断”足够突出，同时又不让文件信息和追溯链路被挤到下方零散区域。

### 6.5.2 模型说明展示规则

纯人声评测结果区内：

- 只展示当前选中模型的说明
- 如果当前选中 `DNSMOS`，就只展示 `DNSMOS`
- 如果当前选中 `NISQA`，就只展示 `NISQA`
- 信号分析作为独立模块说明存在

布局补充规则：

- 单文件页中，当前模型说明和信号分析模块说明建议放在任务进度上方
- 对比页中，当前模型说明和信号分析模块说明也放在任务进度上方
- 单文件页与对比页保持相同位置规则，减少用户切换成本

对比页补充规则：

- 对比页同样展示“当前模型说明 + 信号分析模块说明”
- 建议放在“对比任务进度”上方，便于用户先明确当前判断依据，再看进度和结果

## 6.6 一期不做批量功能

网页端一期先不做独立批量功能。

原因：

- 当前单文件和多文件对比已经能覆盖核心使用需求
- 批量功能会显著拉长结果页和任务结构复杂度
- 现在更重要的是先把场景区分、模型解释、结果追溯和多组对比做扎实

补充约束：

- 顶层非对比主入口明确命名为 `单文件测评`
- 该入口当前只支持单文件上传，不应以“添加文件”这类可能暗示多文件的文案出现
- 未来如果恢复批量能力，需要先更新网页预览和产品文案，再开放入口

但架构和产品结构上必须预留批量扩展空间：

- 工具栏布局预留扩展位
- 结果页结构不写死只能容纳单文件
- 历史页能兼容未来批量任务
- 导出结构能兼容未来批量导出

## 6.7 多组对比

这是后续重点能力之一。

当前已确认规则：

- 输入区支持动态增加对比组
- 初始显示 2 组
- 第 3、4、5 组出现时，后面继续显示 `+`
- 最多支持 6 组

### 6.7.1 对比输入区规则

1. 初始有 A、B 两组输入卡
2. B 后面显示 `+`
3. 点击 `+` 后增加 C
4. C 后面继续显示 `+`
5. 直到 F 为止
6. F 出现后不再显示新增按钮

### 6.7.2 对比结果展示建议

因为 3 到 6 组时，传统 A/B 双栏对比会迅速失效，因此网页端建议采用以下结构：

- 顶部：对比方式切换
- 首屏：对比结论卡
- 首屏：推荐排序卡
- 下部：详细维度 / 信号分析 / 完整表格

推荐的首屏逻辑：

1. 先直接告诉用户“当前推荐哪一组”
2. 如果存在基准组，再说明“相对基准组提升了多少”
3. 再给出 2 到 3 个最值得继续细看的备选
4. 最后再进入完整数据表复核

也就是首屏优先服务决策，不是先把所有组别平权丢进一张大表。

### 6.7.3 基准组逻辑

对比应支持两种模式：

- `自由对比`
- `基准对比`

对应规则：

1. 默认进入 `自由对比`
2. 在没有原文件或参考版本时，直接基于模型结果 + 信号分析做综合排序
3. 如果用户明确知道哪一组是原文件或参考版本，可切换到 `基准对比`
4. 切到 `基准对比` 后，用户可主动选择 A / B / C / D / E / F 中任意一组为基准
5. 结果区所有差值、排序说明和对比标记都跟随基准组切换

适用场景：

- `自由对比`
  - 多个不同参数版本，没有先验参考
  - 目标是先找出当前综合最优版本
- `基准对比`
  - 已知原文件、初版、参考成片或对照版本
  - 目标是输出“相对基准提升/回退”的报告

### 6.7.4 多组对比结果需要包含什么

每组都需要有：

- MOS 结果
- 预处理追溯结果
- 信号分析结果
- 综合排序位置

如果当前处于 `基准对比` 模式，还需要额外包含：

- 与基准组的差值

表头和字段规则也需要随模式切换：

- `自由对比`
  - 展示综合判断
  - 展示排序
  - 不强行展示“相对 A 差值”这类字段
- `基准对比`
  - 展示相对基准差值
  - 展示排序
  - 标签区明确当前基准组

对比总览表至少需要体现以下信号分析对比字段：

- LUFS
- True Peak
- Clipping
- THD

也就是你第 4 条要求在这里必须完全成立。

补充展示建议：

- 表格不再承担首屏决策角色
- 表格更适合作为“详细数据”模块存在
- 建议放在首屏结论卡和推荐排序卡之后
- 页面中可以提供：
  - `模型维度`
  - `信号分析`
  - `完整表格`
  这样的细看入口
- 细看入口应有明确激活态，避免用户分不清当前查看维度

## 7. 综合音频分析页面需求

## 7.1 页面目标

让用户对“混合内容音频”做综合判断，不只看技术指标，还看内容完成度与整体听感。

## 7.2 页面组成

页面应由以下区域组成：

1. 顶部说明区
2. 工具栏
3. 上传区 / 空状态
4. AI 结果区
5. 信号分析区
6. 对比区

内容区补充规则：

- 页头已经显示 `综合音频分析` 和场景说明后，内容区不再重复放一组同义标题
- 内容区直接进入上传区、结果区或对比区
- 工具栏主入口命名应与页面语义一致，使用 `单文件分析` 与 `对比分析`

## 7.3 空状态

空状态应明确告诉用户：

- 这是综合内容页面
- 适合上传什么类型的素材
- 将得到哪两类结果：AI 内容评价 + 信号分析

建议文案：

- 标题：`拖拽综合音频或视频文件到此处`
- 副标题：`适用于人声+音乐、视频音轨、节目成品与混合内容`

## 7.4 单文件结果

单文件结果建议包含两大块：

### 7.4.1 文件与预处理信息

在 AI 结果前先展示：

- 原始文件信息
- 当前使用模型
- 预处理路径
- 送模型前的处理链

建议示例：

- `原始文件 → 抽取视频音轨 → 转单声道 → 保持 48kHz → 送入 AudioBox Aesthetics`

### 7.4.2 AI 内容评价

基于 AudioBox Aesthetics，展示：

- PQ
- CE
- CU
- PC
- 等级
- 简短解释

### 7.4.3 信号分析

必须和纯人声评测页一样完整：

- LUFS
- LRA
- True Peak
- Clipping
- THD

### 7.4.4 综合判断

建议在首屏直接给出一句“适合做什么”的判断，例如：

- `适合发布前微调`
- `内容质量不错，但响度偏高`
- `听感尚可，技术指标需继续优化`

单文件首屏布局建议：

- 与纯人声评测保持一致，使用一张全宽总览卡
- 卡内再分成左右两栏
- 左侧放：
  - 文件名
  - 总分 / 等级
  - 首屏综合结论
- 右侧放：
  - 文件摘要
  - 工程判断
  - 预处理追溯

这样可以维持两个页面的结构复用，同时避免单文件总览只占半行导致视觉失衡。

### 7.4.5 模型说明展示规则

综合音频分析结果区内：

- 只展示当前选中模型 `AudioBox Aesthetics` 的说明
- 信号分析作为独立模块说明存在
- 不在结果页里默认展开其他非当前模型说明

布局补充规则：

- 单文件页中，当前模型说明和信号分析模块说明建议放在任务进度上方
- 对比页中，当前模型说明和信号分析模块说明也放在任务进度上方
- 单文件页与对比页保持相同位置规则

对比页补充规则：

- 对比页同样展示“当前模型说明 + 信号分析模块说明”
- 建议放在“对比任务进度”上方

## 7.5 对比分析

综合音频分析页也应支持多组对比，而不是只支持 A/B。

规则与纯人声评测保持一致：

- 最多 6 组
- 每组均支持 AI 结果 + 信号分析
- 每组均支持预处理追溯
- 结果支持基准组差值、排名和维度对比

对比总览表至少需要体现以下信号分析对比字段：

- LUFS
- True Peak
- Clipping
- THD

综合音频分析页的对比展示同样建议采用分层结构：

- 首屏先给出当前推荐版本
- 再给出推荐排序
- 再提供详细数据表复核

不要直接让用户在首屏面对完整大表寻找结论。

## 8. 所有结果统一支持信号分析

这是网页端非常重要的统一规则。

## 8.1 统一原则

无论用户在哪个页面：

- `纯人声评测`
- `综合音频分析`
- 单文件
- 对比

都应该能看到信号分析结果。

## 8.2 为什么必须统一

如果只有一部分页面有信号分析，会带来这些问题：

- 用户难以理解两个页面真正差异
- 对比时维度不一致
- 历史记录无法统一
- 导出结构会分裂

因此网页端应该从一开始就按“统一结果结构”设计。

## 8.3 网页端统一结果结构

建议结果结构逻辑上至少包含：

1. 文件基础信息
2. 页面场景信息
3. AI / MOS 模型结果
4. 信号分析结果
5. 综合结论
6. 导出信息

无论页面如何变化，这 6 层都不应缺失。

---

## 9. 历史页面需求

## 9.1 目标

历史页面不是简单“存记录”，而是任务回放和复查入口。

历史页不需要重复页头同义说明区块，进入页面后应直接看到任务摘要和记录列表。

## 9.2 列表项应包含

- 时间
- 页面类型
  - 纯人声评测
  - 综合音频分析
- 使用模型
- 文件数量
- 核心摘要
- `查看详情` 入口

历史列表右侧操作建议统一为单一主入口：

- `查看详情`

不要在列表层混用：

- `导出结果`
- `重新查看`
- `查看详情`

这些动作可以统一收进详情页，避免列表区操作语义不一致。

历史页展示建议：

- 列表上方增加筛选条
- 至少支持：
  - 全部
  - 纯人声评测
  - 综合音频分析
  - 自由对比
  - 基准对比
- 右侧增加搜索框，用于按文件名、模型名、日期检索

## 9.3 历史页不应再绑定固定维度

不能再用只适配 `OVRL / SIG / BAK` 的思路来设计历史页。

历史页必须适应：

- DNSMOS
- NISQA
- AudioBox Aesthetics
- 后续新模型
- 信号分析字段

因此历史页应围绕“任务摘要”来设计，而不是围绕某个模型的固定列设计。

## 10. 设置页面需求

设置页建议分为 4 类：

1. 默认模型
2. 结果显示
3. 预处理
4. 导出与历史

设置页不需要重复页头同义说明区块，进入页面后应直接看到配置分组。

页面右上角建议显示版本号，用于预览、测试和结果溯源。

## 10.1 默认模型

- 默认人声模型
- 默认综合音频模型

## 10.2 结果显示

- 是否显示预处理追溯
- 默认对比模式

## 10.3 预处理

- 自动重采样
- 自动转单声道
- 视频自动提取音轨

## 10.4 导出与历史

- 默认导出格式
- 历史保留周期

不建议放进第一期设置页的项：

- 批量功能开关
- 进度展示模式
- 重新分析入口开关

原因：

- 这些不是用户需要频繁调整的业务设置
- 更像实现层策略或后续能力占位
- 放进设置页会增加噪音，但不会提升当前主流程效率

## 11. 操作指南

## 11.1 用户第一次使用

用户进入网页端后，应按以下逻辑理解产品：

1. 先判断素材属于哪一类
   - 如果是纯人声，进 `纯人声评测`
   - 如果是人声+音乐或节目成品，进 `综合音频分析`
2. 进入页面后看顶部说明
3. 上传文件
4. 查看 AI / MOS 结果
5. 查看信号分析结果
6. 决定是否导出或进入对比模式

## 11.2 纯人声评测操作流程

### 单文件

1. 进入 `纯人声评测`
2. 查看顶部说明是否匹配当前素材
3. 上传纯人声音频或纯人声视频
4. 查看 OVRL / SIG / BAK
5. 查看 LUFS / LRA / True Peak / Clipping / THD
6. 根据结果决定是否需要重新处理

### 多组对比

1. 进入 `纯人声评测`
2. 点击 `对比评测`
3. 依次添加 A、B，并按需要继续点击 `+` 增加 C~F
4. 先决定使用 `自由对比` 还是 `基准对比`
5. 如果存在原文件或参考版本，再选择基准组
6. 查看当前模型说明和信号分析模块说明
7. 先看 `对比结论`
8. 再看 `推荐排序`
9. 最后进入 `详细数据` 细看模型维度、信号分析和完整表格
10. 必要时再对照每组预处理追溯信息复核原因

## 11.3 综合音频分析操作流程

### 单文件

1. 进入 `综合音频分析`
2. 上传综合内容素材
3. 查看预处理追溯链路
4. 查看 AudioBox Aesthetics 各维度结果
5. 查看信号分析结果
6. 根据综合判断决定是否适合发布或需继续后期处理

### 多组对比

1. 进入 `综合音频分析`
2. 点击 `对比分析`
3. 添加最多 6 组素材
4. 先决定使用 `自由对比` 还是 `基准对比`
5. 如果存在原始混音、参考成片或对照版本，再选择基准组
6. 查看当前模型说明和信号分析模块说明
7. 先看 `对比结论`
8. 再看 `推荐排序`
9. 最后进入 `详细数据` 查看模型结果、信号分析和完整表格
10. 必要时再回看每组预处理追溯信息

## 11.4 历史记录操作流程

1. 进入 `历史`
2. 按时间、任务类型、模型或关键词筛选
3. 打开历史任务详情
4. 查看当时的 AI / MOS 结果和信号分析结果
5. 选择重新导出或后续重新分析

历史记录里的对比任务还需要额外记录：

- 当时使用的是 `自由对比` 还是 `基准对比`
- 如果有基准组，需要记录具体基准文件

## 12. 导出需求

导出建议至少支持：

- CSV
- JSON

### 12.1 CSV 导出

适合：

- 表格复核
- QA 汇总
- 数据分析

### 12.2 JSON 导出

适合：

- 保存完整结构
- 供后续系统接入
- 保留多模型和信号分析全部字段

### 12.3 导出内容原则

任何导出都应尽量包含：

- 任务类型
- 页面类型
- 文件基础信息
- AI / MOS 结果
- 信号分析结果
- 综合结论

## 13. 交互与视觉规则

## 13.1 总体风格

网页端仍保持：

- 深色专业工具风格
- 玻璃层叠面板
- 清晰的卡片与表格区分

## 13.2 信息优先级

每个结果页都应遵守：

1. 先给结论
2. 再给核心分数
3. 再给信号分析
4. 最后给细节

## 13.3 PC 浏览器原则

网页端当前只服务于 PC 浏览器，不为移动端设计主流程。

因此：

- 可以使用 hover 提示
- 可以使用更高密度的信息布局
- 多组对比可按桌面宽屏优先设计
- 不需要为触屏操作单独设计复杂适配

## 14. 当前已确认项

- 页面命名为：
  - `纯人声评测`
  - `综合音频分析`
- 场景副标题不长期挂在导航上
- 场景说明通过 `hover 提示 + 页面顶部说明` 双层表达
- `综合音频分析` 使用 `AudioBox Aesthetics`
- 页面内需要补充不同模型的优缺点与分析原理介绍
- 结果页只展示当前选中模型说明，不同时展开其他模型说明
- 每次结果都需要展示从原始文件到送模型前的预处理链路
- 进度需要动态逐渐显示，按 `0% → 100%` 和阶段推进
- 网页端一期先不做独立批量功能，但结构上要预留可扩展空间
- 对比输入支持动态 `+`
- 最多支持 6 组
- 对比支持 `自由对比` 和 `基准对比` 两种模式
- 任意组可切换为基准组，但不默认强制选择 A 为基准
- 所有结果都要支持音频信号分析
- `纯人声评测` 与 `综合音频分析` 除模型结果结构外，其余页面项尽量复用
- 网页端只支持 PC 浏览器，不支持移动端
- 页头标题已存在时，内容区不再重复放同义标题和说明
- 对比流程与单文件一样，也必须走完整信号分析链路
- 对比页在任务进度上方展示当前模型说明和信号分析模块说明
- 对比总览表需要展示信号分析对比字段
- 当前处于对比模式时，顶部对比按钮高亮
- 左侧模型区按页面切换，只显示当前页面对应模型组
- 页面右上角显示版本号，便于溯源
- `添加文件` 不作为持续高亮按钮展示

## 17. 当前预览实现对齐

截至 `2026-05-18`，当前网页端预览已经明确采用三层结构：

- `design/web-preview.html`
  页面结构与样式壳
- `design/web-preview-data.js`
  共享业务数据、列定义、格式化与展示映射
- `design/web-preview-app.js`
  页面状态、DOM 渲染与交互逻辑

### 17.1 当前已落地的关键规则

- 顶层主入口目前只按 `web-preview` 确认两类：
  - `单文件测评`
  - `对比评测`
- `单文件测评` 当前只应被视为非对比主入口，并且仅支持单文件上传，不应擅自扩展为独立的“批量结果页”产品流。
- `对比评测` 是当前唯一明确确认的多组对比主流程。
- `详细数据` 已拆成三个独立视图：
  - `模型维度`
  - `信号分析`
  - `完整表格`
- 非完整表格视图不再显示无关列
- `DNSMOS` 与 `NISQA` 已拥有独立对比数据，不再复用同一套分数
- `基准对比` 已不再只是切换标签，而会同步更新：
  - 推荐结论
  - 排序
  - 差值列
  - 基准标签
- `预处理追溯` 开关只隐藏追溯信息，不隐藏历史摘要

### 17.2 当前测试基线

网页端预览当前已有两类测试：

- Node 测试：
  - `tests/web_preview_data.test.mjs`
- Python 测试：
  - `tests/test_web_preview_data.py`
  - `tests/test_web_preview_app.py`
