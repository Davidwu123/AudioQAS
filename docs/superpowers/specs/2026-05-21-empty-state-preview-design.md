# 无假数据初始化预览页设计说明

## 1. 背景

当前 `web-preview` 已经具备：

- `纯人声评测`
- `综合音频分析`
- `单文件`
- `对比`

四类主流程的结果页骨架，以及真实上传后填充运行结果的能力。

但当前预览初始化仍然存在一个明显问题：

- 页面首次进入时会展示假结果
- 部分对比流程在上传单个文件后会自动触发分析

这会带来两个直接问题：

1. 假数据会干扰用户判断当前页面到底有没有真实结果
2. 对比流程缺少“准备输入 -> 明确开始任务”的边界，进度条和任务语义不清楚

本设计说明只解决一个子项目：

**重新设计 `web-preview` 的初始化态，让页面不再依赖假数据，而是按真实结果缓存、空状态、待开始状态和完成状态来显示。**

## 2. 本次范围

本次设计覆盖以下四个页面场景：

- `纯人声评测 / 单文件`
- `纯人声评测 / 对比`
- `综合音频分析 / 单文件`
- `综合音频分析 / 对比`

本次只设计以下内容：

- 初始化空状态
- 对比页待开始状态
- 进度条生命周期
- 失败后的错误态
- 不同模型下真实结果与空状态的切换规则
- 主预览页内的状态分支承载方式

## 3. 明确不在本次范围内的内容

以下内容不属于本次设计：

- 真实后端接口重构
- 历史页改版
- 设置页改版
- 导出流程重构
- 新增批量产品流
- 删除现有结果页骨架
- 移除现有真实结果展示逻辑

本次目标不是重做结果页，而是**重做初始化态和任务启动节奏**。

## 4. 推荐方案

采用“**直接在现有主预览页中增加状态分支渲染 + 现有结果 view 保留**”的方案。

### 4.1 方案内容

- 继续使用当前：
  - `design/web-preview.html`
  - `design/web-preview-data.js`
  - `design/web-preview-app.js`
- 现有结果 view 不删除
- 在主预览页内新增初始化态、待开始态、运行态、完成态和失败态的状态分支
- 所有页面仍然维持一个物理入口，不拆成长期并存的第二个预览页

### 4.2 为什么不新建专用预览页

不采用“新增 `web-preview-empty-state.html` 再回灌主页”的方案，原因有三点：

1. 这会制造短期双页并存，增加后续合并与对齐成本
2. 当前需求本质是状态分支渲染，不是独立产品页
3. 长期来看，一个页面内按状态切换显示更干净，不会形成临时资产残留

### 4.3 为什么不做调试开关

不采用“在现有页面加一个假数据/空状态切换开关”的方案，因为：

- 它是典型过渡性工程垃圾
- 会污染正式预览结构
- 最终仍然需要删除

## 5. 状态模型设计

### 5.1 缓存维度

真实结果缓存按以下 key 维度独立保存：

- `page`
- `scene`
- `model`

即：

- `eval / single / dnsmos`
- `eval / single / nisqa`
- `eval / compare / dnsmos`
- `eval / compare / nisqa`
- `analysis / single / audiobox`
- `analysis / compare / audiobox`

### 5.2 总体显示规则

页面展示只看“当前上下文 key 下是否已有真实结果”：

- 有真实结果：显示该 key 下的真实结果页
- 没有真实结果：显示该 key 下的空状态或待开始状态

切换模型时：

- 不删除其他模型的真实结果缓存
- 只切换当前上下文显示
- 当前模型下有真实结果就显示真实结果
- 当前模型下没有真实结果就显示空状态

也就是说，本次要去掉的是**初始化假数据**，不是**模型间真实结果缓存**。

### 5.3 单文件状态

单文件页使用四态：

- `empty`
- `running`
- `done`
- `error`

含义：

- `empty`：当前 `page + single + model` 下没有真实结果
- `running`：当前正在执行单文件评测或分析
- `done`：当前模型下最近一次真实结果已完成并可展示
- `error`：本次单文件任务失败，需要明确展示失败原因和下一步操作

### 5.4 对比状态

对比页使用五态：

- `empty`
- `ready`
- `running`
- `done`
- `error`

含义：

- `empty`：当前 `page + compare + model` 下没有真实结果，且用户还未准备够对比文件
- `ready`：已上传至少 2 组文件，但还未开始任务
- `running`：当前正在执行完整对比任务
- `done`：当前模型下最近一次真实对比结果已完成并可展示
- `error`：本次对比任务失败，需要明确展示失败原因、保留已准备文件，并允许重试

## 6. 单文件页设计

### 6.1 适用页面

- `纯人声评测 / 单文件`
- `综合音频分析 / 单文件`

### 6.2 保留内容

单文件空状态下仍保留：

- 顶部 toolbar（场景切换入口）
- 模型说明 + 信号分析双卡（`notes-row`）
- toolbar 按钮作为场景切换（`data-scene-trigger`），不直接触发上传

这意味着空状态不是“空白页”，而是“结果主体区替换为任务入口态”。

### 6.3 替换内容

不再在初始化时显示：

- 假的结果概览
- 假的模型结果卡
- 假的信号分析卡
- 假的详细表格
- 假的预处理追溯

已移除的冗余展示元素：

- "查看模型说明"按钮
- "进入多组对比"按钮
- "添加文件"副按钮
- 场景说明文字（"适用于通话..."）
- "当前模型下暂无真实结果"弱提示

### 6.4 单文件空状态结构

主体区改为一个上传卡组件，视觉上与对比页 A/B 上传卡一致：

- 卡组件 `data-single-upload-card`
  - 组名标题
    - `添加文件开始评测`
    - `添加文件开始分析`
  - 上传提示：`点击上传文件`
  - 格式提示：`音频 wav/flac/mp3/aac/m4a/ogg · 视频 mp4/mov/mkv/avi`
  - 当前状态：`未上传`
  - 上传后文件摘要（由 `renderSingleUploadCards` 渲染）：
    - 文件名
    - 文件类型标签
    - 文件大小
    - 状态切换为文件信息

顶部 toolbar 按钮只做场景切换（`data-scene-trigger`），不直接触发上传动作：
- `单文件测评 / 分析`：切换到 single 场景并显示上传卡
- `对比测评 / 分析`：切换到 compare 场景

上传动作由卡片自身的 click 事件触发，不在空状态面板中额外放置"添加文件"按钮。

已移除的冗余元素：
- "查看模型说明"按钮：模型说明已在上方 notes-row 中直接展示，不需要额外滚动按钮
- "进入多组对比"按钮：toolbar 已提供场景切换入口
- "当前模型下暂无真实结果"弱提示
- 场景说明文字（"适用于通话、口播..."等）：与上方模型说明卡内容重复

### 6.5 单文件文案规则

页面类型必须影响文案：

#### 纯人声评测

- 标题：`添加文件开始评测`
- 上传动作目标：执行评测

#### 综合音频分析

- 标题：`添加文件开始分析`
- 上传动作目标：执行分析

避免使用统一的模糊文案，例如 `开始处理`，因为它不能明确区分“评测”和“分析”。

## 7. 对比页设计

### 7.1 适用页面

- `纯人声评测 / 对比`
- `综合音频分析 / 对比`

### 7.2 核心交互原则

对比任务必须从“自动跑”改为“统一开始”：

- 上传文件时不自动触发对比任务
- 用户先准备多组输入
- 再统一点击按钮开始完整任务

原因：

- 对比本质上是一个整体任务，不是多次单文件任务的拼接
- 这样能让进度条只表达“一次完整对比任务”
- 也能避免每加一个文件就重跑一次的混乱体验

### 7.3 empty 状态

空状态下固定先展示两个上传卡：

- `A组`
- `B组`

不采用”先只有一个加号，再逐步生组”的方式，因为对比至少需要两组，先给 `A/B` 更直接。

但必须同时保留并复用当前已有的 `+` 组扩展入口。

也就是说：

- 第一版仍然强引导用户先准备 `A/B`
- 但页面上继续保留现有的 `+` UI
- `+` 的语义明确为”新增对比组”

这样可以直接承接 `C/D/E/F` 的扩展路径，避免后续对 3 组以上对比重新设计输入区。

每个上传卡都必须是可点击上传卡，而不是只有字母标签。

每个卡包含：

- 组名：`A组` / `B组`
- 主文案：`点击上传文件`
- 次文案（格式提示）：`音频 wav/flac/mp3/aac/m4a/ogg · 视频 mp4/mov/mkv/avi`
- 当前状态：`未上传`

上传完成后的文件摘要最少包含（由 `renderCompareUploadCards` 渲染）：

- 文件名
- 文件类型标签
- 文件大小
- 当前状态：
  - `已上传`
  - `已就绪，等待开始`

第一版不强依赖上传前本地解析出的时长、采样率和声道数，以避免把 `ready` 状态依赖到额外的前端媒体解码逻辑上。

上传 2 组文件后 `+` 按钮不自动消失，状态仍停留在 `empty`，直到用户主动点击”开始对比”按钮才转入 `running`。

同时页面保留：

- 对比模式切换
  - `自由对比`
  - `基准对比`
- 主按钮（半透明幽灵风格，不使用高饱和度纯色背景）
  - `开始对比评测`
  - `开始对比分析`

但在少于 2 组文件时，主按钮置灰，并显示提示：

- `至少添加 2 组文件后开始对比评测`
- `至少添加 2 组文件后开始对比分析`

对比模式切换（`自由对比` / `基准对比`）在所有场景下联动更新，使用 `querySelectorAll` 确保同一页内所有 mode-root 实例同步切换。

### 7.4 ready 状态

当用户已上传至少 2 组文件，但还未点击开始时，页面进入 `ready`。

`ready` 不是空状态，也不是结果态，而是独立的待开始预备态。

该状态需要显示：

- 已上传组的文件摘要
- 当前模型
- 当前对比模式
- 沿用现有 `+` 组扩展入口
- 主按钮：
  - `开始对比评测`
  - `开始对比分析`

同时：

- 不显示假的对比 summary
- 不显示假的 ranking
- 不显示假的 detail table

这样用户能明确知道：

- 文件已准备好
- 但任务还没开始
- 当前页面没有伪结果

### 7.5 done 状态

对比完成后，显示当前模型下该 key 的最近一次真实结果。

结果页仍可沿用现有：

- summary
- ranking
- detail table

本次不重做这些结果区，只替换初始化态和启动前态。

### 7.6 error 状态

对比任务失败后，页面进入 `error`。

该状态必须满足：

- 保留已上传的组文件，不自动清空输入
- 显示明确失败原因
- 提供明确的重试动作
- 不伪造新的结果内容

如果当前 key 下此前已有一次成功结果，则失败后应：

- 保留最近一次成功结果可见
- 在顶部显示失败提示：
  - `本次对比失败，当前展示仍为上一次完成结果`

如果当前 key 下此前从未成功过，则失败后应：

- 停留在失败态面板
- 提供：
  - `重新开始对比评测`
  - `重新开始对比分析`
  - 或重新选择组文件

## 8. 进度条设计

### 8.1 总体原则

进度条不能一直常驻显示，必须跟任务状态绑定。

### 8.2 单文件页进度条

#### `empty`

- 不显示进度条

#### `running`

- 显示正式进度条
- 进度来源为真实上传字节追踪（`uploadWithProgress`）：
  - 0%–50%：真实 HTTP 上传字节进度（`XHR.upload.onprogress`）
  - 55%：上传完成，显示"模型评测中"
  - 60%–90%：后端处理阶段，渐进推进
  - 100%：任务完成

#### `done`

- 保留进度条
- 固定显示 `100%`
- 但切换为弱化的完成态组件，而不是继续保持运行中时的强视觉权重
- 标题：
  - `评测完成`
  - `分析完成`

### 8.3 对比页进度条

#### `empty`

- 不显示进度条

#### `ready`

- 不显示正式进度条
- 只显示预备态提示：
  - `已准备 2 组文件，等待开始对比评测`
  - `已准备 2 组文件，等待开始对比分析`

#### `running`

- 显示正式进度条
- 进度来源为真实上传字节追踪（`uploadWithProgress`）：
  - 0%–50%：真实 HTTP 上传字节进度（`XHR.upload.onprogress`）
  - 55%：上传完成，显示"模型评测中"
  - 60%–90%：后端处理阶段，渐进推进
  - 100%：任务完成

#### `done`

- 保留进度条
- 固定显示 `100%`
- 但切换为弱化的完成态组件，而不是继续保持运行中时的强视觉权重
- 标题：
  - `对比评测完成`
  - `对比分析完成`

### 8.4 为什么完成后保留 100%

完成后保留 `100%` 的原因是：

- 能明确告诉用户这次任务已完整结束
- 尤其在对比页中，可以降低“是不是还没算完”的疑惑
- 同时能更自然地承接“当前展示的是该模型下最近一次真实结果”

但完成后的视觉权重必须下降：

- 不再和运行中时一样强强调
- 可以收缩为完成条、完成徽章或弱化版完成模块
- 核心要求是“保留 100% 信息”，不是“保留完整运行中样式”

## 9. 页面与视觉建议

### 9.1 单文件空状态

采用”上传卡组件”方式，视觉与对比页 A/B 上传卡一致：

- 单个居中的 `group-card-upload` 卡片
- 点击即可触发文件选择和上传
- 上传后自动展示文件摘要（文件名、类型、大小）
- 不需要额外的”添加文件”按钮或文字说明

视觉目标：

- 保持现有暗色玻璃风格
- 卡片样式与对比页一致，降低设计不一致感
- 上方保留模型说明 + 信号分析双卡
- 格式提示内嵌在卡片中，不再单独使用 format-group 样式

### 9.2 对比空状态

采用”组卡 + 统一开始”的视觉结构：

- `A/B` 上传卡作为主要视觉焦点
- 对比模式为辅助控制
- 开始按钮在文件准备完成后成为核心动作
- 开始按钮使用半透明幽灵风格（`rgba(88,166,255,0.12)` 背景），避免高饱和度纯色造成的视觉刺眼感
- 对比结论区域与按钮之间保留足够间距（`margin-top:18px`），避免视觉重叠

### 9.3 模型说明与信号分析双卡

空状态和结果态共用同一份双卡布局：

- 左侧：模型说明卡（`data-model-note`），带 `.active-model` 高亮边框
- 右侧：信号分析说明卡（`data-signal-note`），带 `.active-signal` 高亮边框
- 两者共享相同的边框样式，视觉权重一致
- 信号分析内容由 `renderModelContent` 根据 scope 动态填充，右侧不再留空白

### 9.3 页面类型文案必须明确

所有主动作和提示都必须跟页面类型绑定，不能使用模糊总称：

#### 纯人声评测

- `开始对比评测`
- `添加文件开始评测`
- `对比评测完成`

#### 综合音频分析

- `开始对比分析`
- `添加文件开始分析`
- `对比分析完成`

## 10. 前端架构建议

### 10.1 不删除现有结果 view

现有结果 view 保留：

- 结果态仍然使用已有结果页骨架
- 本次只新增空状态和待开始态

### 10.2 新增状态层

建议在前端运行态中按 `page + scene + model` 管理：

- 当前真实结果
- 当前输入准备情况
- 当前状态：`empty / ready / running / done / error`

### 10.3 导出文件命名规则

导出结果文件名必须包含 `request_id` 以区分不同请求：

- 格式：`audioqas_${page}_${scene}_${requestId}.${ext}`
- 例如：`audioqas_eval_single_abc123.json`

### 10.4 主预览页状态分支职责

主预览页需要按状态切换显示：

- `empty`
- `ready`
- `running`
- `done`
- `error`

也就是说，这次改动的实现重点不在”增加第二个页面”，而在”把当前页面的状态分支显式化、可维护化”。

### 10.5 上传链路架构

上传链路使用 `uploadWithProgress` 函数，自动检测运行环境：

- 浏览器环境（XHR `upload` 属性可用）：使用 `XMLHttpRequest` + `upload.onprogress` 追踪真实字节进度
- jsdom 测试环境（XHR `upload` 不可用）：fallback 到 `fetch`，调用 `onProgress(10/50/100)` 模拟进度阶段
- 检测逻辑：`new XMLHttpRequest().upload !== undefined && XMLHttpRequest.prototype.upload !== undefined`

模型切换时 compare 缓存必须保存完整上下文：
- `results`（对比结果 payload）
- `groups`（已上传组文件）
- `base`（基准组选择）
- `requestId`（导出文件名依赖）

### 10.6 error 态与 done 态共存

对比任务失败后，如果当前 key 下已有成功结果：
- done 面板继续显示（不隐藏）
- done 面板顶部加 `data-error-banner` 横幅，提示”本次对比失败，当前展示仍为上一次完成结果”
- error 面板同时显示，提供重试入口

如果当前 key 下从未成功：
- 只显示 error 面板，提供”重新开始”按钮

## 11. 成功标准

本设计完成后，以下条件必须同时满足：

1. 页面首次进入时，不再显示任何假结果
2. 当前 `page + scene + model` 下有真实结果时，仍能直接看到真实结果
3. 切换模型不会删除其他模型的真实结果缓存
4. 对比页上传文件后不会自动开始任务
5. 对比页在至少 2 组文件准备完成前不能开始任务
6. 单文件页与对比页的文案能明确区分”评测”和”分析”
7. 进度条只在 `running / done` 显示
8. 完成后保留 `100%`
9. 对比页沿用现有 `+` UI 作为 3 组以上输入的扩展入口
10. 失败后不会卡在 `running`，而是进入明确 `error` 状态
11. 对比页上传≥2组文件后进入 `ready` 状态，`+` 按钮不消失
12. 模型切换后 compare 缓存恢复完整上下文（results + groups + base + requestId）
13. 对比 error 态下如有上次成功结果，done 面板继续可见并加 error banner
14. 单文件上传入口通过 `data-single-upload-card` 卡片点击触发，toolbar 只做场景切换
15. `uploadWithProgress` 在 jsdom 环境自动 fallback 到 fetch，测试覆盖不受影响
