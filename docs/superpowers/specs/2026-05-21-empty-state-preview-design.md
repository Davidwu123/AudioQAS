# 无假数据初始化预览页设计说明

## 1. 背景

当前 `web-preview` 已经具备：

- `纯人声评测`
- `综合音频分析`
- `单文件`
- `对比`

四类主流程的结果页骨架，以及真实上传后填充运行结果的能力。

但当前预览初始化仍然存在一个明显问题：

- 页面首次进入时会展示假结果
- 部分对比流程在上传单个文件后会自动触发分析

这会带来两个直接问题：

1. 假数据会干扰用户判断当前页面到底有没有真实结果
2. 对比流程缺少“准备输入 -> 明确开始任务”的边界，进度条和任务语义不清楚

本设计说明只解决一个子项目：

**重新设计 `web-preview` 的初始化态，让页面不再依赖假数据，而是按真实结果缓存、空状态、待开始状态和完成状态来显示。**

## 2. 本次范围

本次设计覆盖以下四个页面场景：

- `纯人声评测 / 单文件`
- `纯人声评测 / 对比`
- `综合音频分析 / 单文件`
- `综合音频分析 / 对比`

本次只设计以下内容：

- 初始化空状态
- 对比页待开始状态
- 进度条生命周期
- 不同模型下真实结果与空状态的切换规则
- 新预览页的承载方式

## 3. 明确不在本次范围内的内容

以下内容不属于本次设计：

- 真实后端接口重构
- 历史页改版
- 设置页改版
- 导出流程重构
- 新增批量产品流
- 删除现有结果页骨架
- 移除现有真实结果展示逻辑

本次目标不是重做结果页，而是**重做初始化态和任务启动节奏**。

## 4. 推荐方案

采用“**新预览页承载空状态改版 + 现有结果 view 保留**”的方案。

### 4.1 方案内容

- 新增一个空状态专用预览页，例如：
  - `design/web-preview-empty-state.html`
- 该页仍沿用当前产品结构：
  - `纯人声评测`
  - `综合音频分析`
  - 各自包含 `单文件` 与 `对比`
- 现有结果 view 不删除
- 新页只增加初始化态、待开始态和与真实结果缓存相关的显示规则

### 4.2 为什么不直接改现有主预览页

不推荐直接在当前 `web-preview.html` 上先做这轮初始化改版，原因有三点：

1. 当前结果页已经多轮对齐过视觉和结构，直接改会提高回归风险
2. 这次主要变化集中在初始化和任务启动节奏，不是结果展示层
3. 新预览页更适合单独评审“无假数据初始化”体验，确认后再回灌主页

### 4.3 为什么不做调试开关

不采用“在现有页面加一个假数据/空状态切换开关”的方案，因为：

- 它是典型过渡性工程垃圾
- 会污染正式预览结构
- 最终仍然需要删除

## 5. 状态模型设计

### 5.1 缓存维度

真实结果缓存按以下 key 维度独立保存：

- `page`
- `scene`
- `model`

即：

- `eval / single / dnsmos`
- `eval / single / nisqa`
- `eval / compare / dnsmos`
- `eval / compare / nisqa`
- `analysis / single / audiobox`
- `analysis / compare / audiobox`

### 5.2 总体显示规则

页面展示只看“当前上下文 key 下是否已有真实结果”：

- 有真实结果：显示该 key 下的真实结果页
- 没有真实结果：显示该 key 下的空状态或待开始状态

切换模型时：

- 不删除其他模型的真实结果缓存
- 只切换当前上下文显示
- 当前模型下有真实结果就显示真实结果
- 当前模型下没有真实结果就显示空状态

也就是说，本次要去掉的是**初始化假数据**，不是**模型间真实结果缓存**。

### 5.3 单文件状态

单文件页使用三态：

- `empty`
- `running`
- `done`

含义：

- `empty`：当前 `page + single + model` 下没有真实结果
- `running`：当前正在执行单文件评测或分析
- `done`：当前模型下最近一次真实结果已完成并可展示

### 5.4 对比状态

对比页使用四态：

- `empty`
- `ready`
- `running`
- `done`

含义：

- `empty`：当前 `page + compare + model` 下没有真实结果，且用户还未准备够对比文件
- `ready`：已上传至少 2 组文件，但还未开始任务
- `running`：当前正在执行完整对比任务
- `done`：当前模型下最近一次真实对比结果已完成并可展示

## 6. 单文件页设计

### 6.1 适用页面

- `纯人声评测 / 单文件`
- `综合音频分析 / 单文件`

### 6.2 保留内容

单文件空状态下仍保留：

- 顶部 toolbar
- 当前模型选择
- 页面标题与副标题
- 模型说明卡

这意味着空状态不是“空白页”，而是“结果主体区替换为任务入口态”。

### 6.3 替换内容

不再在初始化时显示：

- 假的结果概览
- 假的模型结果卡
- 假的信号分析卡
- 假的详细表格
- 假的预处理追溯

### 6.4 单文件空状态结构

主体区改为一个强空状态面板，建议包含：

- 标题
  - `添加文件开始评测`
  - `添加文件开始分析`
- 一句简短场景说明
- 支持格式分组
  - `音频：wav / flac / mp3 / aac / m4a / ogg`
  - `视频：mp4 / mov / mkv / avi`
- 主按钮
  - `添加文件`
- 次按钮
  - `查看模型说明`
- 弱提示
  - `当前模型下暂无真实结果`
  - `切换模型后会显示该模型已有结果或空状态`

### 6.5 单文件文案规则

页面类型必须影响文案：

#### 纯人声评测

- 标题：`添加文件开始评测`
- 上传动作目标：执行评测

#### 综合音频分析

- 标题：`添加文件开始分析`
- 上传动作目标：执行分析

避免使用统一的模糊文案，例如 `开始处理`，因为它不能明确区分“评测”和“分析”。

## 7. 对比页设计

### 7.1 适用页面

- `纯人声评测 / 对比`
- `综合音频分析 / 对比`

### 7.2 核心交互原则

对比任务必须从“自动跑”改为“统一开始”：

- 上传文件时不自动触发对比任务
- 用户先准备多组输入
- 再统一点击按钮开始完整任务

原因：

- 对比本质上是一个整体任务，不是多次单文件任务的拼接
- 这样能让进度条只表达“一次完整对比任务”
- 也能避免每加一个文件就重跑一次的混乱体验

### 7.3 empty 状态

空状态下固定先展示两个上传卡：

- `A组`
- `B组`

不采用“先只有一个加号，再逐步生组”的方式，因为对比至少需要两组，先给 `A/B` 更直接。

每个上传卡都必须是可点击上传卡，而不是只有字母标签。

每个卡包含：

- 组名：`A组` / `B组`
- 主文案：`点击上传文件`
- 次文案：
  - `音频：wav / flac / mp3 / aac / m4a / ogg`
  - `视频：mp4 / mov / mkv / avi`
- 当前状态：`未上传`

同时页面保留：

- 对比模式切换
  - `自由对比`
  - `基准对比`
- 主按钮
  - `开始对比评测`
  - `开始对比分析`

但在少于 2 组文件时，主按钮置灰，并显示提示：

- `至少添加 2 组文件后开始对比评测`
- `至少添加 2 组文件后开始对比分析`

### 7.4 ready 状态

当用户已上传至少 2 组文件，但还未点击开始时，页面进入 `ready`。

`ready` 不是空状态，也不是结果态，而是独立的待开始预备态。

该状态需要显示：

- 已上传组的文件摘要
- 当前模型
- 当前对比模式
- 主按钮：
  - `开始对比评测`
  - `开始对比分析`

同时：

- 不显示假的对比 summary
- 不显示假的 ranking
- 不显示假的 detail table

这样用户能明确知道：

- 文件已准备好
- 但任务还没开始
- 当前页面没有伪结果

### 7.5 done 状态

对比完成后，显示当前模型下该 key 的最近一次真实结果。

结果页仍可沿用现有：

- summary
- ranking
- detail table

本次不重做这些结果区，只替换初始化态和启动前态。

## 8. 进度条设计

### 8.1 总体原则

进度条不能一直常驻显示，必须跟任务状态绑定。

### 8.2 单文件页进度条

#### `empty`

- 不显示进度条

#### `running`

- 显示正式进度条
- 阶段建议为：
  - 文件解析
  - 预处理
  - 模型推理
  - 信号分析
  - 结果整理

#### `done`

- 保留进度条
- 固定显示 `100%`
- 标题：
  - `评测完成`
  - `分析完成`

### 8.3 对比页进度条

#### `empty`

- 不显示进度条

#### `ready`

- 不显示正式进度条
- 只显示预备态提示：
  - `已准备 2 组文件，等待开始对比评测`
  - `已准备 2 组文件，等待开始对比分析`

#### `running`

- 显示正式进度条
- 阶段建议为：
  - 文件解析
  - 预处理
  - 模型推理
  - 信号分析
  - 对比整理

#### `done`

- 保留进度条
- 固定显示 `100%`
- 标题：
  - `对比评测完成`
  - `对比分析完成`

### 8.4 为什么完成后保留 100%

完成后保留 `100%` 的原因是：

- 能明确告诉用户这次任务已完整结束
- 尤其在对比页中，可以降低“是不是还没算完”的疑惑
- 同时能更自然地承接“当前展示的是该模型下最近一次真实结果”

## 9. 页面与视觉建议

### 9.1 单文件空状态

建议采用“结果页骨架内嵌强空状态面板”的方式，而不是简单上传框。

视觉目标：

- 保持现有暗色玻璃风格
- 中间主体是任务入口
- 右侧或上方保留模型说明
- 明确“还没有结果”

### 9.2 对比空状态

建议采用“组卡 + 统一开始”的视觉结构：

- `A/B` 上传卡作为主要视觉焦点
- 对比模式为辅助控制
- 开始按钮在文件准备完成后成为核心动作

### 9.3 页面类型文案必须明确

所有主动作和提示都必须跟页面类型绑定，不能使用模糊总称：

#### 纯人声评测

- `开始对比评测`
- `添加文件开始评测`
- `对比评测完成`

#### 综合音频分析

- `开始对比分析`
- `添加文件开始分析`
- `对比分析完成`

## 10. 前端架构建议

### 10.1 不删除现有结果 view

现有结果 view 保留：

- 结果态仍然使用已有结果页骨架
- 本次只新增空状态和待开始态

### 10.2 新增状态层

建议在前端运行态中按 `page + scene + model` 管理：

- 当前真实结果
- 当前输入准备情况
- 当前状态：`empty / ready / running / done`

### 10.3 新预览页职责

新预览页的职责是：

- 专门展示“无假数据初始化”的产品体验
- 允许评审空状态、待开始状态和完成态进度条规则
- 不承担历史兼容或旧预览对照职责

## 11. 成功标准

本设计完成后，以下条件必须同时满足：

1. 页面首次进入时，不再显示任何假结果
2. 当前 `page + scene + model` 下有真实结果时，仍能直接看到真实结果
3. 切换模型不会删除其他模型的真实结果缓存
4. 对比页上传文件后不会自动开始任务
5. 对比页在至少 2 组文件准备完成前不能开始任务
6. 单文件页与对比页的文案能明确区分“评测”和“分析”
7. 进度条只在 `running / done` 显示
8. 完成后保留 `100%`

## 12. 下一步实施方向

本设计稿确认后，下一步 implementation plan 应至少拆成以下几块：

1. 新建空状态预览页
2. 增加单文件 `empty / running / done` 状态渲染
3. 增加对比 `empty / ready / running / done` 状态渲染
4. 改造对比上传交互为“统一开始”
5. 接入按 `page + scene + model` 维度的真实结果缓存与显示切换
6. 补充对应测试
